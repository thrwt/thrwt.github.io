Study Notes

Artifacts

Digital Forensic Artifacts on a Windows system can be classified into two categories

  • System Artifacts: These are related to system activities (e.g., system logs, event logs, registry changes)
  • User Artifacts: These are related to user activity, such as files a user has accessed, browser history, and recent documents.

.lnk File (important)

.lnk files (shortcut files) in Windows contain metadata that links to another file or application. They store information like the target file's location, making it easier to access frequently used files.

LNK Files – Metadata and Signature

  • .lnk files are metadata files interpreted by the Windows Shell.
  • They have a unique file signature (0x4C or "4C 00 00 00"), which can be used for forensic analysis or recovery.

Metadata Found in .lnk Files

  • File path of the target.
  • MAC address of the host computer (sometimes).
  • Last accessed size of the target file.
  • Volume serial number where the target file was stored.
  • Network share name (if applicable).
  • File attributes (read-only, hidden, encrypted, etc.).
  • Distributed link tracking information.

Common Locations of .lnk Files

  • %USERPROFILE%\Recent (Recent Documents folder).
  • %USERPROFILE%\Application Data\Microsoft\Office\Recent (for MS Office shortcuts).

.lnk File Time Stamps

  • .lnk files store their own MAC (Modified, Accessed, Created) timestamps.
  • When a target file is opened, its timestamps are copied into the .lnk file.
  • Windows uses an 8-byte FILETIME format to store timestamps.

Filesystem Timestamps in .lnk Files

  • The creation timestamp of a .lnk file shows the first time the target file was accessed.
  • The modification timestamp indicates the last time it was opened.

Identical Timestamps – Meaning

  • If the timestamps of the .lnk file and the file system match, it means the target file was accessed only once from the shortcut.

No Embedded Dates in Certain Cases

  • Some .lnk files don't store timestamps, especially if they were created when saving a new file.

Microsoft Office Behavior

  • When an Office file is saved for the first time, two .lnk files may be created:
    • One in the "Recent" folder (without embedded dates).
    • One in the "Office Recent" folder (with embedded dates).

Security Risks of LNK Files

Never underestimate an LNK file! It can be configured to execute malicious commands when opened, making it a common method for spreading malware.

ThumbCache / Windows Icon

  • Thumbnail creation: These thumbnails are created when a user views or interacts with a directory containing media files.
  • Thumbnails remain cached even if the original file is deleted, allowing analysts to identify deleted content.
  • Metadata within thumb cache files can include original file paths, helping determine where a file was accessed.
  • Cached thumbnails can reveal what a user has viewed, even if it is later deleted, moved or hidden.

Locate the files: %USERPROFILE%\AppData\Local\Microsoft\Windows\Explorer

TOOL: Thumbcache Viewer

Mapping the Image

  • There is a method to map images using the Extensible Storage Engine (ESE)
  • The database file used is called Windows.edb, which contains indexed data from the system.

Stored in C:\ProgramData\Microsoft\Search\Data\Applications\Windows

ShellBags

  • Shellbags are Windows registry artifacts that log user interactions with folders even after those folders are deleted.
  • They contain timestamps and metadata that help reconstruct user activity timelines.
  • These registry keys record metadata such as folder paths, view settings and timestamps

Where Are ShellBags Stored?

User-Specific Shellbags

  • HKEY_USERS\<SID>\Software\Microsoft\Windows\Shell\BagMRU
  • HKEY_USERS\<SID>\Software\Microsoft\Windows\Shell\Bags

Current User Shellbags

  • HKEY_CURRENT_USER\Software\Microsoft\Windows\Shell\BagMRU
  • HKEY_CURRENT_USER\Software\Microsoft\Windows\Shell\Bags

Supporting Files

Shellbag data is also found inside user registry hive files:

  • NTUSER.DAT: Located at C:\Users\<Username>\NTUSER.DAT
  • USRCLASS.DAT: Located at C:\Users\<Username>\AppData\Local\Microsoft\Windows

Tool: ShellBags Explorer – Extracts and analyzes folder history.

Volume Shadow Copy Service - VSS

VSC (Volume Shadow Copy)

A set of COM APIs that implements a framework to allow volume backups to be performed

Mechanism

  1. VSC tracks changes to blocks of data on the disk rather than logical files in their entirety
  2. When you mount the VSC, it will appear as a mirror of the whole contents of the volume that was monitored.

Storage Limitation

  • Windows automatically manages snapshots, deleting older ones when storage is full.
  • Windows limits the maximum disk space consumed by VSC snapshots is 5% of the volume size.

Snapshot Creation Triggers

  • Created automatically during updates, driver installations, scheduled tasks, or manually.

Forensic Value

  • Helps recover deleted files, registry keys, logs, and tampered data for investigation.

Tools for Accessing VSC

  • Vssadmin + mklink (Windows built-in tool)
  • ShadowCopyView, Shadow Explorer (GUI tools)
  • Libvshadow, VSC Toolset (Forensic tools)

File History (Windows 8+)

  • Backs up user data (Libraries, Desktop, Contacts, etc.) hourly.
  • Uses 5% of disk space for offline caching and retains backups indefinitely.
  • Stores cached files in: %USERPROFILE%\AppData\Local\Microsoft\Windows\FileHistory\Data

Forensic Insights from File History

  • Backup directories, user ID, PC name, retention policy, storage location

Windows Recycle Bin

Older Windows Versions (Before Vista)

Deleted files were renamed using the format: C<DriveLetter><Index#>.<FileExtension> (e.g., DC2.exe).

Metadata was stored in the INFO2 file, containing:

  • Original filename & path
  • File size
  • Deletion date/time

Newer Versions of Windows

Starting from Windows Vista, the structure changed. Instead of a single INFO2 file, Windows now creates two separate files for each deleted file in the $Recycle.Bin directory:

  1. $I<ID_String>.<extension> → Metadata file
    • Stores information such as: Original file path, original filename, file size, date and time of deletion.
  2. $R<ID_String>.<extension> → Raw file content
    • Contains the actual deleted file's data.

These files are named using a unique ID string to prevent conflicts.

Tools for Analyzing $Recycle.Bin

1. Rifiuti & Rifiuti-Vista

  • Used to parse and analyze $I files (metadata files).
  • Command to analyze $Recycle.Bin metadata:
.\rifiuti-vista.exe -x -z "C:\$Recycle.Bin\S-1-5-21-3459457667-847189511-749124871-1001\"
  • -x → Outputs XML format.
  • -z → Enables timezone conversion.

2. $I_Parse (GUI Tool)

  • A graphical tool for extracting metadata from $I files.
  • Displays original filename, path, deletion date, and size
NOTE: the last digits in the ID are the same digits in the ID users — you can check it in the registry.

Prefetch Files (evidence of execution)

Prefetch files (.pf) are small files automatically created by Windows when an executable file is run. Their main purpose is to help Windows launch programs faster.

Windows uses them to speed up application launching by tracking:

  • The first time the program ran.
  • The last 8 times it ran.
  • Files and DLLs it accessed

Mechanism

  • Windows Cache Manager tracks:
    • First 2 minutes of system boot.
    • First 10 seconds of application startup.
  • The data is stored in .PF (Prefetch) files under C:\Windows\Prefetch.

Prefetch File Contents

Each .PF file contains:

  1. Executable name
  2. Absolute path to the executable
  3. Number of times the program has run
  4. Last execution timestamp
  5. List of DLLs used by the program

Registry Settings (Enable/Disable Prefetching)

If no prefetch files exist, check:

HKLM\SYSTEM\CurrentControlSet\Control\SessionManager\MemoryManagement\PrefetchParameters
  • 0 → Disabled (No prefetching)
  • 1 → Application Prefetching Only
  • 2 → Boot Prefetching Only
  • 3 → Both Application & Boot Prefetching

Tool for Prefetch Analysis

  • PECmd (Prefetch Explorer Command-line Tool)
    • Extracts and analyzes prefetch files.
    • Example command:
PECmd.exe -d "D:\0x\The-Hood\prefetch" --csv D:\ --csvf pf.csv

Program Compatibility Assistant (important for execution)

A Windows feature that monitors applications for issues like crashes, compatibility problems, or suspicious behavior.

Microsoft Protection Log (MPLog)

MPLog stands for Microsoft Protection Log, a plain-text log generated by Windows Defender

  • Process execution tracking – they can show which processes ran and when
  • Detection events – logs include records of threats detected, such as malware signatures, file paths, process IDs

C:\ProgramData\Microsoft\Windows Defender\Support\

ShimCache (Application Compatibility Cache) AppCompatCache

  • ShimCache helps Windows track applications that need compatibility settings to run properly on newer versions of Windows.
  • It provides backward compatibility for older software.

Forensic Value

ShimCache stores:

  1. Executable/script file names & full paths
  2. Whether the file actually ran or was just viewed in Explorer
  3. Last modified timestamp (Standard Information Last Modified Date)
  4. Size of the binary file
💡 Important: Even if an executable isn't run, just browsing through Windows Explorer will cause it to be logged in ShimCache. If more executables appear after resizing the Explorer window, they will also be shimmed but only saved after a reboot or shutdown.

Locations

  1. Registry Key: 
    HKLM\SYSTEM\CurrentControlSet\Control\SessionManager\AppCompatCache\AppCompatCache
  2. SYSTEM Hive File: 
    C:\Windows\System32\Config\SYSTEM

Tool for Analysis

  • AmcacheParser (Parses and extracts useful forensic data from ShimCache).
  • AppCompatParser

Windows Registry

Reference: Uncovering Windows Forensics Artefacts for Digital Forensics Investigators

1. User Activity

🔸 RecentDocs (NTUSER.DAT)

  • Location: HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs
  • Tracks most recently accessed files by extension and path.
  • Forensics value: Shows what documents were accessed and when.

🔸 TypedURLs / TypedPaths

  • Location:
    • URLs: HKCU\Software\Microsoft\Internet Explorer\TypedURLs
    • Paths: HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths
  • Shows manual inputs in the address bar (web or folder).
  • Forensics value: Can reveal intentional user navigation to sensitive locations or websites.

🔸 UserAssist

  • Location: HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist
  • Tracks programs executed by the user.
  • Values are ROT13-encoded (e.g., HRZR_EHACngU = UEM_RUNPath).
  • Forensics value: Precise insight into what the user launched, including timestamps (last execution time).

🔸 ShellBags

  • Location: Stored in NTUSER.DAT and UsrClass.dat
  • Tracks folder views, including deleted folders and network shares.
  • Forensics value: Shows access to paths no longer present, useful for identifying hidden or removed folders.

🖥️ 2. System Configuration & Timeline

🔸 System Boot and Shutdown

  • Keys like HKLM\SYSTEM\ControlSet001\Control\Windows → ShutdownTime (FILETIME format).
  • LastBootUpTime can be pulled from Event Logs or registry volatility plugins.
  • Forensics value: Timeline anchoring (helps sequence events before/after reboots).

🔸 MountedDevices

  • Location: HKLM\SYSTEM\MountedDevices
  • Shows drive letters and corresponding device GUIDs or serials.
  • Forensics value: Detects previously connected USBs or external drives even if they're not currently attached.

🔸 Time Zone Settings

  • Location: HKLM\SYSTEM\CurrentControlSet\Control\TimeZoneInformation
  • Forensics value: Ensures accurate timeline interpretation—important for correlating with logs and timestamps.

🔸 Network Interfaces

  • Location: HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces
  • IP addresses, DHCP lease times, etc.
  • Forensics value: Determines past network connections and configurations.

🧑‍💻 3. User Accounts

🔸 SAM Hive

  • Location: C:\Windows\System32\config\SAM
  • Contains user account names, RIDs, and password hash references.
  • Forensics value: User enumeration, privilege level tracking.

🔸 NTUSER.DAT

  • Per-user hive found in C:\Users\[username]\
  • Holds user-specific settings, app history, typed commands, etc.
  • Forensics value: User attribution, activity, and persistence mechanisms.

🔸 ProfileList

  • Location: HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList
  • Links SID to profile path and LastUseTime.
  • Forensics value: Confirms presence of users and last interaction.

🧬 4. Malware and Persistence

🔸 Run / RunOnce Keys

  • Locations:
    • HKCU\Software\Microsoft\Windows\CurrentVersion\Run
    • HKLM\SYSTEM\CurrentControlSet\Services\...
  • Commonly abused to maintain persistence across reboots.
  • Forensics value: Detects automatic execution of malware or suspicious scripts.

🔸 AppInit_DLLs

  • Location: HKLM\Software\Microsoft\Windows NT\CurrentVersion\Windows
  • Allows DLLs to be loaded into every user-mode process.
  • Forensics value: High stealth method for persistence.

🔸 Services

  • Location: HKLM\SYSTEM\CurrentControlSet\Services
  • Malware may install itself as a service to start with the system.
  • Forensics value: Look for non-default services or odd paths.

🔸 ShimCache (AppCompatCache)

  • Location: HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache
  • Stores metadata of executed programs, even if deleted.
  • Forensics value: Proof of execution even without Prefetch or logs.

🛠️ 5. Software & Application Artifacts

🔸 Installed Software

  • Location: HKLM\Software\Microsoft\Windows\CurrentVersion\Uninstall
  • Lists all installed programs with install date, version, and sometimes install path.
  • Forensics value: Tracks suspicious or unauthorized software installations.

🔸 MRU Lists

  • Found throughout NTUSER.DAT (e.g., for Paint, Word, dialog boxes).
  • Tracks last used items like files, folders, and recent registry inputs.
  • Forensics value: Evidence of direct user interaction with specific files.

🔸 Prefetch Paths

  • Registry references often match with prefetch file entries.
  • Used together to confirm program execution.

The Windows Registry is a critical database that stores system and user configurations.

Windows Registry Structure

  • 🗂 Hives – The top-level containers that store registry keys and values.
  • 📂 Keys – Similar to folders, they contain subkeys and values.
  • 📁 Subkeys – Nested inside keys, they hold further subkeys or values.
  • 📄 Values – Store actual data (e.g., settings, configurations).

Browser Forensics

Browser forensics involves analyzing web browsing history, cached data, cookies, and other artifacts stored by web browsers.

Browser Artifact

  1. Browsing History
    • Stores visited websites, including timestamps and URLs.
    • Can help reconstruct a user's online activity.
    • Useful for tracking suspicious or unauthorized browsing behavior.
  2. Cache
    • Stores locally downloaded copies of web content such as images, scripts, and pages.
    • Helps speed up page loading by avoiding repeated downloads.
    • Can contain remnants of deleted web pages, which is useful in forensic investigations.
  3. Cookies
    • Small text files stored by websites to maintain user sessions and preferences.
    • Can contain login details, authentication tokens, and tracking data.
    • Helps in identifying user interactions with websites.

Internet Explorer - Registry and File Locations

  1. Autocomplete Data (Saved Forms & Typed URLs)
    • HKCU\Software\Microsoft\Internet Explorer\Intelliforms\Storage1 — Autocomplete
    • HKCU\Software\Microsoft\Internet Explorer\Intelliforms\Storage2 — Typed URLs
    • Stores user-inputted data in forms and previously typed URLs.
    • Useful for retrieving login credentials and frequently visited sites.
  2. Cache Files
    • %USERPROFILE%\AppData\Local\Microsoft\Windows\Temporary Internet Files\
    • Contains cached web pages, images, and scripts.
    • Can reveal details of visited websites even after the browsing history is cleared.
  3. Bookmarks (Favorites)
    • %USERPROFILE%\Favorites
    • Stores saved websites, allowing investigators to see sites of interest to the user.
  4. Cookies
    • %USERPROFILE%\AppData\Local\Microsoft\Windows\Cookies\Low
    • Stores website cookies, which can help track user behavior, logins, and online services accessed.

Master File Table (MFT)

MFT / J / logfile

File System Journaling

A transactional record of all changes made to a volume.

  • In the event of a crash or power failure, the system uses this information to roll back the changes.

$UsnJrnl

It keeps a log of changes like file creations, deletions, renaming happening on the disk.

Whenever a file is created, deleted, renamed, or modified, the system records it in $UsnJrnl.

It is stored in $Extend$UsnJrnl on an NTFS drive.

Inside it, there are two important streams:

  • $J → The main journal that records file changes.
  • $Max → Controls the maximum size of the journal.

Used by backup/anti-virus software to monitor changes to files

$LogFile

  • Special file in NTFS that keeps a record of low-level file system changes.
  • It tracks detailed operations that happen to the file system.
  • It is stored at the root of an NTFS volume (partition).
  • It ensures file system integrity, meaning it helps recover lost data if something goes wrong (like a system crash).

Unlike $UsnJrnl, which tracks high-level changes (like file creations and deletions), $LogFile tracks deeper system-level changes.

Some examples of operations recorded:

  • AddIndexEntryAllocation → When a new file entry is added to the index.
  • InitializeFileRecordSegment → When a new file record is created.
  • DeleteIndexEntryAllocation → When a file entry is removed from the index.

Example:

  • $UsnJrnl → It records what happened:
    • "File X was created"
    • "File Y was renamed"
    • "File Z was deleted"
  • $LogFile (machine's internal memory) → It tracks how the factory changed things:
    • "Allocated space here"
    • "Changed this file record here"
    • "Moved internal structure from A to B"

Example:

  • $UsnJrnl: Logs → "User created hello.txt"
  • $LogFile: Logs → "NTFS added a new entry to the MFT" → "Allocated cluster space for data" → "Updated directory structure"

SRUM (System Resource Usage Monitor)

The SRUM is a Windows component that logs system activity related to resource usage, primarily for diagnostics and performance tracking. It can be useful in digital forensics because it tracks application usage over time

C:\Windows\System32\sru\SRUDB.dat
  • Tracks executed processes — even if logs or prefetch are cleared.
  • Correlates user activity (through SIDs).
  • Timeline reconstruction of app usage.
  • May show malware persistence or stealthy execution patterns.

SRUDB is a JET Blue (ESENT) database, and it stores data across multiple tables, each recording a different aspect of system activity.

Persistence Mechanisms

Run and RunOnce Registry Keys

Use Run or RunOnce registry keys to make a program run when a user logs on. The Run key makes the program run every time the user logs on, while the RunOnce key makes the program run one time, and then the key is deleted. These keys can be set for the user or the machine.

HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run
HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\RunOnce
HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run
HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\RunOnce

Persistence using GlobalFlags in Image File Execution Options – Hidden from Autoruns.exe

Reference: oddvar.moe — GlobalFlags persistence technique

This technique creates persistence in Windows by abusing two registry features:

  • Image File Execution Options
  • Silent Process Exit
When a specific program exits, Windows automatically launches another program.

The attacker adds three registry values. It can evade startup detection tools.

The persistence may not appear in common startup tools.

  1. Enable monitoring with GlobalFlag in 
    HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\notepad.exe
    Change value GlobalFlag to GlobalFlag = 512 — that means FLG_MONITOR_SILENT_PROCESS_EXIT, telling Windows to monitor when this process exits.
  2. Configure Silent Process Exit 
    HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\SilentProcessExit\notepad.exe
    Change to ReportingMode = 1 — this enables an action when the monitored process exits.
  3. Define the payload — Same registry key, set to: 
    MonitorProcess = C:\temp\evil.exe
    Now Windows knows what program to launch.
