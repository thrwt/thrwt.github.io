Can you recover a deleted file using nothing but its MFT record? Most forensic tools do it for you. But understanding how it actually works is what separates using a tool from understanding the filesystem.

I've been researching NTFS internals and documenting the complete process of analyzing MFT records at the hex level, with a focus on manual deleted file recovery.

Deleted vs Active: The First Check

This is the first check in any deleted-record scan: read 2 bytes at offset 0x16 and mask the low bits.

Offset 0x16 (2 bytes, little-endian)
bit 0 (0x0001): 1 = record in use, 0 = record not in use (deleted)
bit 1 (0x0002): 1 = directory, 0 = file

ValueMeaning
0x00Deleted file
0x01Active file
0x02Deleted directory
0x03Active directory

When you look at offset 0x16, that's how you know: in this case the value reads as an active file.

MFT Slack Space, In Close

MFT slack is the unused space inside an allocated 1024-byte MFT record that contains leftover data no longer referenced by NTFS. It exists in two forms:

  • Record Slack
  • Attribute Slack

Record Slack vs Attribute Slack

Record Slack: the bytes between UsedSize (0x18) and AllocatedSize (0x1C, normally 1024) — at the end of the MFT record. Contains leftover data from previous attributes or older versions of the record.

  • FF FF FF FF at 0x198 is the $END marker terminating the attribute list; everything after it, up to AllocatedSize, is slack.
  • The non-zero run starting at 0x1A8 (31 40 00 00, 0C 00 00 00, B0 00 00 00) reads like fragments of an older, larger attribute header — the values line up with typical attribute-length and content-offset fields.

Attribute Slack: unused bytes between the logical end of an attribute's content and the allocated space for that attribute — inside an attribute. Contains leftover bytes from older content within the same attribute, such as a longer filename or a previous resident $DATA payload.

Deleted Files in NTFS – What Actually Changes

NTFS never shrinks the physical MFT record.

When a File Is Deleted

  • $MFT Bitmap: the record's allocation bit is cleared (0 = unallocated), marking the slot available for reuse.
  • MFT record flags (0x16): bit 0 cleared → 0x00 or 0x02.
  • Updates UsedSize to reflect the new, smaller logical content.
  • Parent directory index ($INDEX_ROOT/$INDEX_ALLOCATION): the directory entry referencing the file is removed.
  • Link count (0x12): decremented; reaches 0 when the last hard link is deleted.
  • $LogFile: records the delete as a transactional log entry.
  • $UsnJrnl ($J): appends a USN_RECORD with USN_REASON_FILE_DELETE.
  • Writes a new $END marker (0xFFFFFFFF) immediately after the last valid attribute, and leaves the remaining bytes unchanged.

These leftover bytes persist until they are overwritten.

What Can Remain in Slack

  • Entire $FILE_NAME attributes from before a rename
  • Entire $STANDARD_INFORMATION blocks with pre-timestomp timestamps
  • Resident $DATA content (small files) — potentially full file recovery from slack alone
  • $OBJECT_ID (GUIDs), $SECURITY_DESCRIPTOR residues, $INDEX_ROOT fragments from directories
  • Fragments of $ATTRIBUTE_LIST entries referencing extension records
  • Old sequence numbers, old link counts, old flag bytes — useful for timeline reconstruction

After Record Reuse

  • Increments the Sequence Number (offset 0x10) to invalidate stale file references.
  • Writes new attributes starting at the First Attribute Offset, overwriting only the bytes needed.
  • Leaves the remaining bytes unchanged, so older data can survive in the record slack if it isn't overwritten.

Recovering Deleted Files

Immediately after deletion, the MFT record is mostly unchanged. Only allocation metadata is updated, while the record's attributes remain intact. Recovery quality decreases only as the record is reused. Broadly, recovery falls into a few scenarios:

  • Recovering Resident Files
  • Recovering Non-Resident Files
  • Recovering Fragmented Files
  • Overwritten Records — recovery here reconstructs the new file, not the deleted one.
  • Partially Overwritten Records
    • Bytes [0 .. new_UsedSize) belong to the new file.
    • Bytes [new_UsedSize .. old_UsedSize_of_prior_content) may still contain fragments of the old file's later attributes — for example, if $STANDARD_INFORMATION and $FILE_NAME were overwritten by the new file's own copies of those same attribute types, but the old $DATA attribute that came after them is still sitting untouched further into the record.

Recovering Resident Files

If the $DATA attribute is resident, the file's content is stored directly inside the MFT record.

The following is a deleted resident MFT record (SetMace64.exe) — identified as deleted because its flags read 0x0000. The record also carries a Zone.Identifier alternate data stream, starting at 0xDD58, alongside the file's own $DATA attribute:

Resident Alternate Data Stream (ADS)

├── $DATA (Named)     Resident
│      Type = 80 00 00 00
│      Name = Zone.Identifier
OffsetBytesMeaning
0xDCA880 00 00 00$DATA attribute
0xDC1600 00Flags = 0x0000 (deleted)
0xDC1004 00Sequence Number = 4 — this MFT entry has been allocated to four different files over its lifetime.
0xDD5880 00 00 00Attribute Type = $DATA — the Zone.Identifier ADS begins here.
0xDD5C98 00 00 00Attribute Length = 152
0xDD6000Non-Resident flag — 0 = resident (content stored inline in this record), 1 = non-resident (content stored elsewhere on disk, referenced via data runs)
0xDD6218 00Name Offset — offset (from the start of this attribute) where the name string begins; used to read the stream name :Zone.Identifier

Recovering Non-Resident Files

This is a deleted non-resident MFT record (Non-Resident Flag = 0x01), meaning the file content is stored outside the MFT record. Rather than containing the file itself, the attribute stores a run list describing the physical disk location of the data. Let's walk through an example.

Suppose the file is thrwt.txt. The MFT record does NOT
contain the file content. Instead it contains something like:

$DATA
Run List
  LCN = 5309
  Length = 1 cluster

That's all. Think of it as a note saying:
"The file is over there."

To find the non-resident data, Windows looks at Run List Offset = 0x40:

0x40 (64) → absolute 0xE908 + 0x40 = 0xE948

Jumping to 0xE948, we find 21 01 BD 14 00. These five bytes are not file data — they're an address, encoded as a data run.

Decode the run header byte, 0x21. Split into nibbles: 2 | 1:

  • Low nibble (length field size) = 1 byte — the run length is the next byte, 01, meaning 1 cluster.
  • High nibble (offset field size) = 2 bytes — the run offset is the next two bytes, BD 14, which in little-endian reads as 14 BD, equal to 0x14BD = 5309. Since this is the first run in the list, its offset is relative to LCN 0, so the starting LCN is 5309.

The final byte, 00, is the terminator marking the end of the run list.

The decoded run list means:

Read 1 cluster starting at LCN 5309

MFT Record
│
├── $DATA (Non-resident)
│
└── Run List
      │
      ▼
LCN 5309
+----------------------+
|                      |
|   File Data          |
|                      |
|                      |
+----------------------+

Recovering Fragmented Files

A fragmented file consists of multiple non-contiguous data runs. The MFT still doesn't contain the file — it contains a map telling Windows where each piece of the file is located.

Suppose the file is thrwt.exe.

The MFT record does NOT contain
<installer data>

Instead it contains something like:

$DATA
Run List: Run 1, Run 2, Run 3

Windows first locates the run list. Run List Offset = 0x40, so the attribute content begins at 0xC918 + 0x40 = 0xC958, where it finds:

21 01 97 14
31 28 41 EB 0B
32 B5 3C BB 2D F4
00

These bytes are not file data — they're simply a map describing where each fragment of the file is stored.

Decode Run #1 (same steps as the non-resident example, just more runs):

21 01 97 14 → header 0x21 splits into nibbles 2 | 1:

  • Length field = 1 byte — next byte 01 means 1 cluster.
  • Offset field = 2 bytes — next two bytes 97 14, little-endian, read as 14 97, equal to 0x1497 = 5271. Since this is the first run, its offset is relative to LCN 0, so the starting LCN is 5271.

Repeating the same steps for each remaining data run recovers the rest of the fragments. From the second run onward, the offset field is a signed delta relative to the previous run's LCN (encoded in two's complement when negative), not an absolute address — that's how a fragment can jump backward on disk.

Run 1: 21 01 97 14
LCN = 5271, Length = 1 cluster

Run 2: 31 28 41 EB 0B
Header 0x31 → length field 1 byte (28 = 40 clusters), offset field 3 bytes (41 EB 0B LE = 781121, added to the previous LCN 5271) → LCN = 786392, Length = 40 clusters

Run 3: 32 B5 3C BB 2D F4
Header 0x32 → length field 2 bytes (B5 3C LE = 0x3CB5 = 15541 clusters), offset field 3 bytes (BB 2D F4 LE, a negative two's-complement delta of -774725 from the previous LCN 786392) → LCN = 11667, Length = 15541 clusters

So: read 1 cluster starting at LCN 5271, then 40 clusters starting at LCN 786392, then 15,541 clusters starting at LCN 11667, and concatenate them in this order to reconstruct thrwt.exe.

That's the whole story — you're able to recover data without any tool, just by reading the bytes.
